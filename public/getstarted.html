Contribute To Our Product

“Team Player” is an open source product, so all of our source code and releases are facilitated through Github. We provide source code for the latest stable released version.

Prior releases can be found on the same page. The latest version of our repository that we’re using for development can be found on our github page.

Build Instructions

The following instructions assume that you are using a CSE Linux machine.

If you are unfamilar with Ruby on Rails, here is a quick guide to the file structure.

  1. First, setup rails environment for your OS
    https://www.ruby-lang.org/en/installation/

  2. Install git, then navigate to folder you want to store our code and run
    git clone https://github.com/panjiw/Team-Player.git

  3. Navigate to the Team-Player folder and run
    bundle install -- without production

  4. To setup the database, run
    bundle exec rake db:migrate

  5. Finally, you can start the rails server by running
    rails s

  6. If successful, you can see our application on
    http://localhost:3000/
    Now you can start editing the codes and view the effects on the localhost website!

  7. For security reasons, you should create your own key of secret.yml for production. To do so, run rake secret
    and copy that key. Go to config/secret.yml and replace your key with the key already there
    youwill have to paste it in two places.

Test Instruction

The tests provided for the software are in the spec folder. At the moment, these tests cover our backend implementation, and utilize RSpec. After you have downloaded our application, to run the test suite provided, go to the root directory of the project, run the command: bundle exec rspec spec/

This will run all the tests and output the results. For more information about RSpec, go to: http://rubydoc.info/gems/rspec-rails/frames

Daily Builds and Tests Set Up

To set up an automated daily build and test, developers will need to find a system to run the test suite above in automation. To run automated daily builds and tests, users will need to run “bundle exec rspec spec/” and “rake jasmine” in automation.

We deployed our daily builds and tests through Jenkins windows installations.
  1. Set up a new project workspace with the home directory as the project directory. We followed the windows installation guide for Jenkins to do this.
  2. Set the product to deploy daily through. We are currently using a local host, which is subjected to change.
  3. We set up email notifications when the build fails.
  4. In order to rub our test suite, we built the product with a windows batch command with the test automation command bundle exec rspec spec/
  5. For front-end tests, we will use another windows batch command of rake jasmine
  6. And to run ruby through Jenkins, we needed to add ruby’s PATH file to Jenkin’s environment.

Feedback from our daily tests can be viewed on our mailing group.

New Version Releases

Releases will be facilitated through GitHub’s release system.

Updates and fixes, as well as branch information and version numbers will be documented with each release.

Releases for our software can be found and downloaded from: https://github.com/panjiw/Team-Player/releases.

New releases will be deployed through Heroku web services.

  1. Be sure to have the heroku toolbelt installed and set up with your account. Follow the directions on the page after you've downloaded it.
  2. For the first deployment, the developer should run
    heroku create
  3. Then, when a new release is completed and ready to be copied to the website and viewed and run through our test suite, a developer can deploy their repository with
    git push heroku master
  4. Be sure to update the requirements, design, user's, and developer's documents on the github after every release.

File Structure

The directory structure uses the standard Ruby on Rails structure. From the root directory, listed below is information for the folders relevant to developers. Other folders contain information necessary for the product, such as bundle and rails files, but not necessary to edit for development.

  • root folder : contains gems and rake files for Ruby

  • app/ : contains the application code, our backend ruby code and front end views.

  • app/models/ : contain our ruby backend data models

  • app/controllers/ : our ruby controllers, which provide functions for frontend to interact with the data

  • app/view/ : our static html pages, the subfolder layout contains shared code like header

  • app/assets/javascript/ : contains all of our frontend AngularJS system:

  • app/assets/javascript/controllers/ :

  • config/ : contains the environment and key configuration for the product.

  • db/ : contains the backend data model schema and migrations files.

  • development/ : Contains our development website HTML/CSS code

  • public/ : contains public HTML documents (I.E. 404 page)

  • spec/ : contains the testing information for backend and frontend development. Relevant tests for each section will be in its own subfolder, I.E. controller RSpec tests being in the controller subfolder and front-end javascript test being in the javascripts subfolder.

Documentation for the code can be found from the README in the main directory of our repository.

Design Patterns

Front End Design Pattern: Model-View-Controller

We use MVC because it makes designing and coding easier because we are able to split up our project based on the pages or views we are working on. We can then further split it based on the functionality. This is helpful because our product will automatically update based on user input, which naturally implies a MVC design. So, data can be read by the controllers, updated by the models, and displayed on the views.

Our main models are the ruby files, within \app\models\, that update state:
  • bill.rb
  • bill_actor.rb
  • group.rb
  • membership.rb
  • task.rb
  • task_actor.rb
  • user.rb
Our main views are html and css files, within \app\views\static_pages\:
  • _bills.html.erb
  • _groups.html.erb
  • _tasks.html.erb
  • _home.html.erb
  • bills.css
  • groups.css
  • home.css
  • login.css
  • tasks.css
Our controllers are javascript files within \apps\assets\javascripts\:
  • billsViewController
  • groupsViewController
  • homeViewController
  • loginViewController
  • mainViewController
  • tasksViewController

Back End Design Pattern: Factory

We use the factory pattern because it allows us to create objects of a same type, given restrictions. The taskGenerator object is a "factory" of tasks that handles the problem of creating new tasks. The front end uses the factory to make a post request to the taskGenerator so it can get the next task from the factory. The factory then provides a task object based on some repeat/cycle logic as well as initialization parameters stored in the database. For now, we only have part of the repeat/cycle implemented, so the factory abstracts out the "special" classes of tasks from the front end, to allow the front end to display the tasks regardless of what kind they are.

Here are the files used in the factory design pattern:
  • \app\models\task_generator.rb
  • \app\models\task_generator_actor.rb
  • \app\controllers\task_generators_controller.rb

Publishing To Heroku

If you plan to put the web app to Heroku, you would need to:

  1. Set up a Heroku account
  2. Install Heroku toolbelt. Then follow the simple instructions on the page.
    • Be sure to run heroku create in the Team-Player folder to create a Heroku app on your account. You will see "confirmation git@[some_name].git
  3. Verify the configuration git remote -v You'll see a (fetch) and (pull) for the [some_name] above.
    • If you already have an app on your account, run
      heroku git:remote -a [some_name]
  4. Deploy the code to heroku
    git push heroku master
  5. Initialize the database
    heroku run rake db:migrate
  6. You can view the site at https://[some_name].herokuapp.com
See the setting up RoR on Heroku page for more information.
