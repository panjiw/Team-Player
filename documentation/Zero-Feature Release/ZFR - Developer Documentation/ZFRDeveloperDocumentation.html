TeamPlayer Developer Documentation

        All of our source code and releases will be facilitated through Github. Source code for the latest stable released version can be found from our github repository: https://github.com/panjiw/Team-Player/releases. Prior releases can be found on the same page. The latest version of our repository that we’re using for development can be found from https://github.com/panjiw/Team-Player.git.

What is the layout of your directory?

        The directory structure is created for Ruby on Rails applications, so we have the standard Ruby on Rails structure. From the root directory, listed below is information for the folders relevant to developers. Other folders contain information necessary for the product, I.E. bundle and rails files, but not necessary to edit for development. For further information on what those files contain, see the above Ruby on Rails structure link.

  • root folder: contains gems and rake files for Ruby
  • app: contains the application code, our backend ruby code and front end views. The subfolders include: assets, controllers, helpers, mailers, and models. These contain code relevant to each of those subsections. For example, the model subfolder contains models for our user data, and the view folder contains the main front-end web files.  
  • config: contains the environment and key configuration for the product.
  • db: contains the main backend data files in the database.
  • devfiles: Contains front end development code. Includes Calendar front end  code, style sheets, and html view pages.
  • public: contains public HTML documents (I.E. 404 page)
  • spec: contains the testing information for backend and frontend development. Relevant tests for each section will be in its own subfolder, I.E. controller RSpec tests are in the controller subfolder and front-end javascript tests are in the javascripts subfolder.

Documentation for the code can be found from the README in the main directory.

How do I build?

The following installation instructions assume you are using a Linux system.

1. First setup rails environment for your OS

https://www.ruby-lang.org/en/installation/

2. Install git, then navigate to the folder you want to store our code and run

“git clone https://github.com/panjiw/Team-Player.git”

3. Navigate to the Team-Player folder and run

        “bundle install -- without production”

4. Note: For security reasons, secrets.yml and database.yml are not shown publicly

on github. To create your own, navigate into the config folder.

        - rename the secrets-example.yml to secrets.yml

        - rename database-example.yml to database.yml

5. To generate your secret_token, run “rake secret.”

                - copy the key, and open secret.yml

                - replace the “run_rake_secrete_and_copy_key_here” message with

your copied key.

- You will have to paste in two places

6. To set up the database, run

“bundle exec rake db:migrate”

7. Finally, you can start the rails server by running “rails s”

8. If successful, you can see our application on http://localhost:3000/

                       Note: If you plan to put the web app to heroku, you would need to:

  • Have heroku toolbelt installed and set up with your account, see https://toolbelt.heroku.com/
  • Run “heroku create” in the Team-Player folder
  • Push the repo up to heroku using “git push heroku master”
  • Initialize the database using “heroku run rake db:migrate“
  • See  https://devcenter.heroku.com/articles/getting-started-with-rails4        for more  info on setting up RoR on Heroku.

How do I test?

The tests provided for the software are in the spec folder. At the moment, these tests cover our backend implementation, and utilize RSpec. More information on our testing system is in the Software Architecture Document. In order to create and run tests, the developer will need to follow the build process to acquire their own database key.

To run the test suite provided, in the directory of the project, run the command “bundle exec rspec spec/” from root folder.

For testing Angular and front-end development, we will be running Jasmine tests through an automation tool called Karma. For this release, we mainly only have static HTML pages, so we do not currently have any front-end tests. We will update this documentation as tests become available.

How do I set up an automated daily build and test?

To set up an automated daily build and test, developers will need to find a system to run the test suite above in automation. To run automated daily builds and tests, users will need to run “bundle exec rspec spec/” and “rake jasmine” in automation.

We deployed our daily builds and tests through Jenkins windows installations.

1) Set up a new project workspace with the home directory as the project directory. We

followed the windows installation guide for Jenkins to do this.

2) Set the product to deploy daily through. We are currently using a local host, which

is subjected to change.

3) We set up email notifications when the build fails.

4) In order to rub our test suite, we built the product with a windows batch command

with the test automation command “bundle exec rspec spec/”.

5) For front-end tests, we will use another windows batch command of “rake jasmine”. 6) And to run ruby through Jenkins, we needed to add ruby’s PATH file to Jenkin’s

environment.

Feedback from our daily tests can be viewed on the mailing group.

How do I view bugs?

Bug tracking is available through our github repository. Bug fixes can be submitted a fix via a commit comment. For example, to fix bug #42 in a commit message the developer would comment with “Fixes #42”. This will close the bug when the commit is merged into the default branch. More information on Github bug fixes can be found here.

How do you release a new version of TeamPlayer?

        Releases will be facilitated by GitHub’s release system. In-depth information on releasing software can be viewed here. When releasing a new version of software, updates and fixes will be documented with the release. Additionally, branch information and version numbers will also be documented with the release.

        Releases for our software can be found and downloaded from here.  New releases will be deployed through Heroku web services. For the first deployment, the developer should call “heroku create”. Then, when a new release is completed and ready to be copied to the website and viewed and run through our test suite, a developer will deploy their repository with “git push heroku master”. More information on how Heroku deploys releases can be found here.

What design patterns do you utilize?

In the front end, the design pattern we use is Model-View-Controller. This helps us make designing and coding easier because we were able to split up our project based on the the page or view we were working on, then further split it up based on the functionality. This is helpful because our product updates based on user input, so data and functions can be passed around through the MVC components.
