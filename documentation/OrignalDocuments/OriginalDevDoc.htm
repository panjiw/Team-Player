How can I obtain the source code?

All of our source code and releases will be facilitated through Github. Source code for

the latest stable released version can be found from our github repository: https://github.com/panjiw/Team-Player/releases. Prior releases can be found on the same page. The latest version of our repository that we’re using for development can be found from https://github.com/panjiw/Team-Player.git.

 

What is the layout of your directory?

The directory structure is created for Ruby on Rails applications, so we have the

standard Ruby on Rails structure: http://www.tutorialspoint.com/ruby-on-rails/rails-directory-structure.htm. From the root directory, listed below is information for the

folders relevant to developers. Other folders contain information necessary for the product,

I.E. bundle and rails files, but not necessary to edit for development. For further information on what those files contain, see the above Ruby on Rails structure link.

● root folder: contains gems and rake files for Ruby

● app: contains the application code, our backend ruby code and front end views. The subfolders include: assets, controllers, helpers, mailers, and models; contain code relevant to each of those subsections, I.E. the models for our user data will be stored in the model subfolder. The view folder contains the main front-end web files.

● config: contains the environment and key configuration for the product.

● db: contains the main backend data files in the database.

● devfiles: Contains front end development code. Includes Calendar front end

code, style sheets, and html view pages.

● public: contains public HTML documents (I.E. 404 page)

● spec: contains the testing information for backend and frontend development. Relevant tests for each section will be in its own subfolder, I.E. controller RSpec tests being in the controller subfolder and front-end javascript test being in the javascripts subfolder.

 

Documentation for the code can be found from the README in the main directory.

 

 

How do I build?

To build, follow these instructions:

1. First setup rails environment, download correct installer for your OS

http://railsinstaller.org/en for the ruby installer

2. Navigate to folder you want to store our application and run

“git clone https://github.com/panjiw/Team-Player.git”

3. Go inside Team-Player folder, and run “bundle install -- without production”

Note: For security reasons, secrets.yml and database.yml are not shown

publicly on github, follow step 4 to create those files.

4. Go to config folder:

+ rename the secrets-example.yml to secrets.yml

+ rename database-example.yml to database.yml.

5. Run “rake secret” to generate your secret_token, copy the key, and paste it

into secrets.yml. Replace “run_rake_secret_and_copy_key_here” with your

key.

6. Run, “bundle exec rake db:migrate” to setup database

7. Run “rails s” to start the server. Click here for more Ruby on Rails

information.

8. If successful, you can see our application on http://localhost:3000/

 

Note: If you plan to put the web app to heroku, you would need to:

● Have heroku tollbelt installed and set up with your account, see

https://toolbelt.heroku.com/

● Run “heroku create” in the Team-Player folder

● Push the repo up to heroku using “git push heroku master”

● Initialize the database using “heroku run rake db:migrate“

● See https://devcenter.heroku.com/articles/getting-started-with-rails4

for more info on setting up RoR on Heroku.

 

 

How do I Test?

The tests provided for the software are in the spec folder. These tests cover our backend implementation, and utilize RSpec. More information on our testing system is in out Software Architecture Document. In order to create and run tests, the developer will need to follow the build process to acquire their own database key.

To run the test suite provided, in the directory of the project, run the command

“bundle exec rspec spec/” from root folder, and to get the test version of the website running, please follow the build instructions above.

For testing Angular and front-end development, we will be running Jasmine (http://www.tutorialspoint.com/ruby-on-rails/rails-directory-structure.htm) tests

through an automation tool called Karma (http://www.tutorialspoint.com/ruby-on-rails/rails-directory-structure.htm). For this release, we mainly only have static HTML pages, so we do not currently have any front-end tests. We will update this documentation as tests become available.

 

 

How do I set up an automated daily build and test?

To set up an automated daily build and test, developers will need to find a system to

run the test suite above in automation. Therefore, when running their own automated daily

build and test, users will need to run “bundle exec rspec spec/” and “rake jasmine” in

automation.

We deployed our daily builds and tests through Jenkins windows installations. After

following the windows installation guide for Jenkins (https://devcenter.heroku.com/articles/releases), we set up a new project workspace with the home directory being the project directory. We set the product to deploy daily through, currently, a local host which is subjected to change. Additionally, we set up email notifications when the build fails. In order to rub our test suite, we built the product with a windows batch command with the test automation command “bundle exec rspec spec/”. For front-end tests, we will use another windows batch command of “rake jasmine”. And to run ruby through Jenkins, we needed to add ruby’s PATH file to Jenkin’s environment.

Feedback from our daily tests can be viewed on the mailing group (https://devcenter.heroku.com/articles/releases).

 

 

How do I view bugs?

Bug tracking is available through our github repository. Bug fixes can be submitted a

fix via a commit comment. For example, to fix bug #42 in a commit message the developer

would comment with “Fixes #42”. This will close the bug when the commit is merged into the default branch.

 

 

How do you release a new version of TeamPlayer?

Releases will be facilitated by GitHub’s release system. In-depth information on

releasing software can be viewed at https://help.github.com/articles/creating-releases. When releasing a new version of software, updates and fixes will be documented with the release. Additionally, branch information and version numbers will also be documented with the release.

Releases for our software can be found and downloaded from  https://github.com/panjiw/Team-Player/releases. New releases will be deployed through Heroku web services. For the first deployment, the developer should call “heroku create”. Then, when a new release is completed and ready to be copied to the website and viewed and run through our test suite, a developer will deploy their repository with “git push heroku master”. More information on how Heroku deploys releases can be found from https://devcenter.heroku.com/articles/releases.
