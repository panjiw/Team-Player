Since our application has a lot of features we expected our customers to have problems understanding each feature and how to use them. However, to our surprise, the customer was able to understand and use our features pretty easily after seeing them for the first time. We also expected them  to spend more time on the task creation process, using our cycle and repeating features, but they skipped those features quite quickly. We expected the user to understand that all of the features revolved around groups and working as a team, but the user seemed to consider the calendar as an individual task organizer.

We did not expect confusion however right after login, we might need to add a help or intro page after a new account have been made. We will fix this by automatically placing users in their own group, and having helpful directions to guide users to create groups, tasks, and bills. In addition to that, we also did not expect that users would want to click on blank space in the calendar, which might also be useful. We saw that it was intuitive for users to want to create a task or bill from a certain day, so we will allow task/bill creation from the detailed view of the day of the calendar, instead of restricting them to buttons next to the calendar. Also, our bill display made sense to us, but wasn’t useful to our user. Instead of seeing all bills before seeing payment totals, we will format bill viewing to seeing these totals first, which can be filtered and manipulated. Lastly, we had forgotten to centralize groups during task and bill creation - we had kept it to simply adding people. We will add group selection that automatically selects all members in that group, and the user can deselect members for a task or bill.

Something we could have done differently to make the exercise go better is we could have spent time showing the backend group our design and paper prototype and letting them run through it before showing the customers. They were not completely familiar with it when we presented so they could not answer all of the customers’ questions. I think this is good to keep in mind as we start working on the actual code because we are going to have to communicate back and forth between backend and frontend to make sure that the backend can provide the information we need for the frontend.
