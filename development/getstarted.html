Want to contribute to our product?

Prior releases can be found on the same page. The latest version of our repository that we’re using for development can be found from here https://github.com/panjiw/Team-Player.git.

Getting started

If you are unfamilar with Ruby on Rails, here is a quick guide to the file structure The following installation instruction assume you are using a Linux system.

  1. First setup rails environment for your OS
    https://www.ruby-lang.org/en/installation/

  2. Install git, then navigate to folder you want to store our codes and run
    git clone https://github.com/panjiw/Team-Player.git

  3. Navigate inside Team-Player folder and run
    bundle install -- without production

  4. Note: For security reasons, secrets.yml and database.yml are not shown publicly on github
    To create your own navigate inside config folder
    -- rename the secrets-example.yml to secrets.yml
    -- rename database-example.yml to database.yml

  5. To generate your secret_token, run
    rake secret
    copy the key, and open secret.yml, replace the "run_rake_seret_and_copy_key_here" message
    with your copied key, you have to paste it in two places

  6. To setup the database run
    bundle exec rake db:migrate

  7. Finally, you can start the rails server, run
    rails s

  8. If successful, you can see our application on
    http://localhost:3000/

Now you can start editing the codes and view the effects on localhost website!

Test Instruction

The tests provided for the software are in the spec folder. At the moement, these tests cover our backend implementation, and utilize RSpec. After you have downloaded our application, to run the test suite provided, go to the root directory of the project, run the command
bundle exec rspec spec/
This will run all the tests and output the results, more information about RSpec follow this link http://rubydoc.info/gems/rspec-rails/frames

Release new verion

Releases will be facilitated by GitHub’s release system. In-depth information on releasing software can be viewed on
https://help.github.com/articles/creating-releases.
When releasing a new version of software, updates and fixes will be documented with the release. Additionally, branch information and version numbers will also be documented with the release.
Releases for our software can be found and downloaded from
https://github.com/panjiw/Team-Player/releases.
New releases will be deployed through Heroku web services. For the first deployment, the developer should call “heroku create”. Then, when a new release is completed and ready to be copied to the website and viewed and run through our test suite, a developer will deploy their repository with “git push heroku master”. More information on how Heroku deploys releases can be found on
https://devcenter.heroku.com/articles/releases.

File Structure

The directory structure is created for Ruby on Rails applications, so we have the standard Ruby on Rails structure. From the root directory, listed below is information for the folders relevant to developers. Other folders contain information necessary for the product, I.E. bundle and rails files, but not necessary to edit for development. For further information on what those files contain, http://www.tutorialspoint.com/ruby-on-rails/rails-directory-structure.htm

  • root folder: contains gems and rake files for Ruby
    • app: contains the application code, our backend ruby code and front end views. The subfolders include: assets, controllers, helpers, mailers, and models; contain code relevant to each of those subsections, I.E. the models for our user data will be stored in the model subfolder. The view folder contains the main front-end web files.
    • config: contains the environment and key configuration for the product.
    • db: contains the main backend data files in the database.
    • devfiles: Contains front end development code. Includes Calendar front end code, style sheets, and html view pages.
    • public: contains public HTML documents (I.E. 404 page)
    • spec: contains the testing information for backend and frontend development. Relevant tests for each section will be in its own subfolder, I.E. controller RSpec tests being in the controller subfolder and front-end javascript test being in the javascripts subfolder.
  • Documentation for the code can be found from the README in the main directory.

Publishing To Heroku

Our application will differentiate itself by providing a calendar view of events to facilitate time-dependant tasks. While HomeSlice provides a lot of services for roommates to organize their group, it does not provide a calendar view that could make the application difficult for paying bills on time or completing tasks on time. Furthermore, HomeSlice specifically targets roommates as a user base and not just groups who need to organize events. Seeing as our application works with general tasks and bills and not specifics like supplies, we can reach a wider audience.
